On the `inAppPurchasesV2` question flagged in the Task 4 code review: the spec does **not** expose an independent `inAppPurchasesV2` resource. `/v1/apps/{id}/inAppPurchasesV2` and `/v1/apps/{id}/relationships/inAppPurchasesV2` exist as sub-paths of `apps`, and the independent `inAppPurchases` resource has a single GET-by-id operation. The current skip set is therefore correct at its current granularity — but see the next section for why that granularity is itself inadequate.

## ⚠ Critical finding — skip granularity is too coarse

The skip list entry `apps` removes **86 operations** because Apple uses the parent resource as a hub for a sprawling set of unrelated sub-path endpoints: analytics, beta feedback, background assets, search keywords, webhooks, and dozens of relationships accessors. The hand-written `AppsService` covers exactly three of them (`GET /v1/apps`, `GET /v1/apps/{id}`, `PATCH /v1/apps/{id}`). The remaining **83 `apps` sub-path operations disappear silently** when skip is applied.

The same pattern holds for `builds` (30 ops vs. the handful `BuildsService` exposes), `appStoreVersions` (30 ops), `betaGroups` (21 ops), and `subscriptions`-via-`subscriptionGroups` relationships.

**Implication:** if we ship Phase 6.1 → 6.2 with the current skip-by-resource-name policy, the generated SDK silently drops a large chunk of real Apple functionality. The design-doc's one-to-one mapping of "skip entry ↔ hand-written service" assumed resources were leaf-level — they are not.

This must be resolved before Phase 6.2 starts emitting code. Two viable options (see also _Open questions_ below):

1. **Tighten skip granularity to `(resource, path_prefix)` pairs.** Example: skip only `/v1/apps` and `/v1/apps/{id}` for the `apps` resource; let the generator emit everything else. Requires a minor file-format bump in `skip.txt` and a parser change to filter per-operation instead of per-resource.
2. **Auto-carve sub-path endpoints into their own generator outputs.** The parser keeps grouping by `/v1/<resource>`, but the emitter splits sub-path operations into helper types distinct from the hand-written service. Cleaner long-term, bigger Phase 6.2 scope.

Either way, the current Phase 6.1 skip count (**20 resources → 251 operations hidden**) is what Phase 6.2 starts from, not what Phase 6.2 accepts as the final policy.

## Operation-naming observations

### Canonical paths work well

The fast-path rules (`GET /v1/<r>` → `List`, `GET /v1/<r>/{id}` → `Get`, `POST` → `Create`, `PATCH` → `Update`, `DELETE` → `Delete`) hit the top-of-resource shapes cleanly across the whole spec. Spot-checks on `appCategories`, `appEvents`, and `subscriptions` look idiomatic:

```
List      GET  /v1/appCategories
Get       GET  /v1/appCategories/{id}
Create    POST /v1/appEvents
Delete    DELETE /v1/subscriptions/{id}
```

### Fallback naming is readable but verbose

52.9% of operations fall back to `verb + PascalCase(path-tail-after-resource)` with `{id}` segments stripped (the Task 8 critical fix). Examples from the real spec:

| Path | Fallback name |
| --- | --- |
| `GET /v1/apps/{id}/customerReviews` | `GetCustomerReviews` |
| `GET /v1/apps/{id}/relationships/appInfos` | `GetRelationshipsAppInfos` |
| `DELETE /v1/apps/{id}/relationships/betaTesters` | `DeleteRelationshipsBetaTesters` |
| `GET /v1/apps/{id}/metrics/betaTesterUsages` | `GetMetricsBetaTesterUsages` |
| `GET /v1/subscriptions/{id}/appStoreReviewScreenshot` | `GetAppStoreReviewScreenshot` |

The good: all generated names are valid Go identifiers and carry enough context to disambiguate.

The awkward:
- `Relationships` is a pure structural prefix that inflates every relationship accessor. Phase 6.3 could collapse `GetRelationshipsFoo` → `GetFooRelationships` or expose these via a dedicated `.Relationships()` sub-type.
- `metrics` is itself a path segment, not a resource, so `GetMetricsBetaTesterUsages` reads awkwardly compared to `GetBetaTesterUsagesMetric`. Phase 6.3 can decide whether to reorder.
- A handful of Apple sub-paths include `V2` (e.g. `appAvailabilityV2`, `appStoreVersionExperimentsV2`, `inAppPurchasesV2`); these survive cleanly as `GetAppAvailabilityV2`, `GetAppStoreVersionExperimentsV2`, etc. The `V2` suffix is preserved by `naming.PascalCase` because it has no acronym carve-out — good outcome, but worth being explicit.

## Unexpected path shapes

- **Deep sub-paths at `/metrics/…`:** `/v1/apps/{id}/metrics/betaTesterUsages` is the only place a 3-deep sub-path is exposed. Current parser folds this into `apps`, naming becomes `GetMetricsBetaTesterUsages`. Fine for now.
- **Bare `/relationships/…` accessors are ubiquitous:** 44 of `apps`' 86 operations are `relationships/…` paths. They are GET-only for most, but `DELETE /v1/apps/{id}/relationships/betaTesters` and `PATCH /v1/apps/{id}/relationships/promotedPurchases` exist — the generator needs to emit mutator variants, not just listers.
- **No `/v1/{...}/...` path (leading path-param) patterns in the spec:** the defensive branch in `resourceFromPath` is confirmed dead code against real input. Leave it for robustness against future spec changes.
- **Empty `components.schemas` items:** a handful of schemas resolve to empty objects (placeholders referenced but never fleshed out). Phase 6.2 should warn rather than emit zero-field types.

## `log.Printf` malformed-path diagnostic stayed silent

The Task 8 diagnostic `parser: skipping malformed path item %q` produced zero output against the real spec. Every path item unmarshalled cleanly into `rawPathItem`. The log path is working infrastructure, not dead code — it just didn't fire because Apple's spec is well-formed at the path-item level.

## Parser TODOs for Phase 6.2+

- **Granular skip policy.** Replace resource-name skip with operation-level skip. Proposed: `skip.txt` gains an optional `resource /path/prefix` form, e.g. `apps /v1/apps` + `apps /v1/apps/{id}` to drop only the 3 operations the hand-written `AppsService` implements. Without this the generator cannot coexist with hand-written services and ship complete coverage. (See _Critical finding_ above.)
- **Schema extraction.** `components.schemas` (1,337 entries) is currently ignored. Phase 6.2 reads it to populate `ir.Resource.Attrs`, `ir.Relationship`, request/response `ir.Type`. This is where the JSON:API `Resource[T]` type machinery lives.
- **`$ref` resolution.** Apple uses `$ref` heavily inside `components.schemas`. The resolver must handle cycles (many relationships point back to the parent app type).
- **`parameters` propagation.** Each operation's `parameters` array currently drops on the floor. Should populate `ir.Operation.PathParams` and `QueryParams` with typed entries.
- **Request/response body typing.** `ir.Operation.RequestBody`/`ResponseBody` are nil today. Phase 6.2 must populate these from the operation's `requestBody` and `responses['200' or '201'].content.application/json.schema`.
- **Naming refinement (Phase 6.3 input).** The `Relationships…` prefix and `Metrics…` awkwardness above. Plus: consider whether sub-path operations should live on the parent service (flat) or on nested sub-types (ergonomic but more surface).
- **`Deprecated` and `x-apple-available-since` mining.** Deprecated ops make it into IR but nothing tests that they render with Go `// Deprecated:` comments yet. Apple also uses `x-apple-*` vendor extensions — Phase 6.2 should survey them.

## Open questions for Phase 6.2 planning

1. **Which skip-granularity option do we take — path-prefix tuples or sub-type auto-carving?** This gates every other Phase 6.2 design choice because it determines what the generator emits for resources with large hand-written coverage.
2. **How do we render JSON:API `relationships/…` accessors?** Flat method on the parent service (current approach) vs. a nested `Relationships()` accessor sub-type. The second reads better at call-sites but doubles the generated-symbol count.
3. **Do we ship any `apps` sub-path operations in Phase 6.2, or carve them out as Phase 6.4 follow-up?** 83 operations is a lot to scope into the first real-generation phase. A phase cut might be: 6.2 ships gen'd resources that are fully outside `skip.txt`, 6.4 re-introduces the sub-paths under the new skip policy.
4. **Should the spec download be CI-gated?** Today `scripts/update-spec.sh` is a manual maintainer ritual. Apple ships spec updates without semver-style versioning; a drift check on every CI run would catch silent spec changes. (Outside Phase 6.2's scope, but worth flagging.)
